Eng-05 · Threat Model STRIDE

Mapa de Ameaças — Vetores Reais do Código

Categoria Ameaça Vetor no Código Mitigação
Spoofing Forjar identidade de qualquer usuário JWT_SECRET='dev_secret' em docker-compose. Segredo trivialmente reversível por brute-force ou uso do fallback hardcoded. Gerar segredo mínimo 256 bits com openssl rand -base64 64. Remover fallback. Rotação automática via Vault.
Tampering Manipular dados de qualquer org GET /sales/leads sem guard — companyId vem de req.user?.companyId || 'default-company-id'. Sem token, acessa org padrão. Reativar @UseGuards(JwtAuthGuard) e vincular companyId do token, nunca do body.
Repudiation Negar ações por audit trail corrompida AuditInterceptor lê req.user?.companyId que nunca existe no payload JWT atual. Todos os logs registram orgId: null. Incluir orgId no JWT payload em signIn. Validar presença de orgId antes de salvar AuditLog.
Info Disclosure Exfiltrar dados de todas as organizações getMyOrg() chama findFirst() sem where. Retorna a primeira org do banco para qualquer usuário autenticado. Filtrar por membership do usuário: where: { members: { some: { userId: req.user.id } } }.
DoS Exaurir créditos Gemini de todos os tenants Throttler global 100 req/min sem distinção por tenant. Endpoints /ai/generate e /playbooks/generate não têm rate limit por org. Throttler por tenant via @Throttle({ default: { ttl: 60, limit: 10 } }) com key baseada em orgId.
Elevation Receber créditos sem pagamento POST /stripe/webhook sem verificação de stripe-signature. Qualquer corpo com type: "invoice.paid" aciona a lógica de billing. stripe.webhooks.constructEvent(rawBody, signature, endpointSecret) — obrigatório antes de qualquer lógica de negócio.

Bloco II — CMO Estrategista

Engenharia Reversa do Modelo de Negócio

O código não mente. Tudo abaixo foi extraído diretamente do repositório — schema, middlewares, constantes e integrações.

CMO-01

Quem Somos — Identidade Institucional

Nossa Essência

Uma plataforma SaaS B2B multi-tenant de prospecção comercial com IA. O código revela isso inequivocamente: entidade Organization como raiz de tudo, RBAC com OWNER/ADMIN/MEMBER, Wallet por org (não por usuário) e integrações com HubSpot, Twilio e Stripe. Não é um CRM genérico — é uma ferramenta de execução de vendas com inteligência contextual embutida.

DNA Tecnológico

NestJS + PostgreSQL + pgvector = "Construídos para durabilidade e memória semântica." A presença de LangChain, embeddings 1536-dim e BullMQ revela um produto que aprende com os dados de comunicação do time comercial. React 19 + Zustand + Framer Motion: UX responsiva pensada para SDRs que trabalham em ritmo acelerado.

CMO-02

O Que Resolvemos — Problema & Solução

O Problema

  • Times comerciais perdem 40–60% do tempo em tarefas manuais: registro de chamadas, busca de contexto antes de ligações, elaboração de mensagens repetitivas.
  • Leads dispersos em planilhas, WhatsApp, CRM e e-mail. Zero visibilidade de contexto no momento da abordagem.
  • Cada chamada não registrada é receita que evaporou sem rastreamento — revelado pelos campos content e metadata em Communication.
  • SDRs sem playbook personalizado para cada vertical perdem conversões que poderiam ser automáticas.

A Solução

  • Unificação omnichannel: voz (Twilio), WhatsApp, e-mail e webhooks em uma única linha do tempo por contato.
  • Memória semântica: base de conhecimento indexada com pgvector — o contexto certo no momento certo, gerado por Gemini.
  • Playbooks gerados por IA: PlaybookController cria sequências de 5 passos personalizadas por indústria e objetivo, cacheadas para reuso.
  • Billing por consumo real: Wallet/Ledger garante que cada minuto de voz e token de IA gera receita mensurável.
CMO-03

Quem Atendemos — ICP e Perfis de Usuário

B2B Confirmado — Evidências do Código

  • Subscription vinculada a Organization, não a User → pagador é empresa, não indivíduo.
  • RBAC com OWNER/ADMIN/MEMBER por workspace → gestão de equipe, não uso individual.
  • Campos department_id ausentes — sinal de SMB, não Enterprise maduro.
  • PLAN_LIMITS.ENTERPRISE.seats: -1 (ilimitado) → há intenção de atender grandes operações.

3 Perfis de Usuário — Extraídos do RBAC

  • OWNER: Fundador ou Head de Vendas. Configura a org, gerencia billing, vê relatórios gerenciais.
  • ADMIN: Gerente de SDR. Cria playbooks, gerencia base de conhecimento, supervisiona times.
  • MEMBER: SDR/Closer. Acessa leads, faz chamadas, envia mensagens, consulta playbooks.

Conformidade & LGPD / Privacidade

Controles Presentes ✓

  • Segmentação por orgId em todas as entidades de negócio — isolamento de dados entre tenants.
  • AuditLog com índice orgId + createdAt para investigações.
  • PiiRedactionService com regex para CPF e cartão de crédito.
  • Helmet + CORS configurável + ZodValidationPipe — headers e input sanitizados.
  • Criptografia de campos sensíveis no HubSpot (AES — algoritmo a corrigir para GCM).

Lacunas Críticas LGPD ✗

  • Sem soft-delete em Contact e Communication — exclusão é permanente e imediata.
  • Sem campo data_retention_days — dados de comunicação e auditoria crescem indefinidamente.
  • Sem endpoints de DSR (Data Subject Request) — portabilidade e exclusão não implementadas.
  • Sem política de consentimento registrada no schema — base legal de tratamento não documentada no código.
  • E-mail, telefone e LinkedIn não são redatados pelo PiiService em textos livres enviados à IA.

Rascunho de Política de Privacidade — Para Revisão Jurídica

  • Dados coletados: Nome, e-mail, telefone, LinkedIn URL (tabela Contact); transcrições e metadados de chamadas, mensagens WhatsApp e e-mails (tabela Communication); dados de billing e consumo (Wallet/Ledger).
  • Base legal: Execução de contrato (Art. 7º, V da LGPD) — dados necessários para prestação do serviço de prospecção comercial contratado.
  • Compartilhamento com terceiros: Google Gemini (conteúdo de prompts e documentos), HubSpot (e-mail + nome + notas), Twilio (número de telefone + áudio de chamadas), Stripe (e-mail + dados de billing).
  • Retenção: A definir — lacuna de compliance atual. Recomendação: Comunicações: 2 anos; Audit logs: 5 anos; Dados de billing: 7 anos (obrigação fiscal).
  • Direitos do titular: Acesso — a implementar via GET /users/me/data; Portabilidade — a implementar via GET /users/me/export; Exclusão — a implementar via DELETE /users/me com cascata.
  • Canal DPO: [privacy@suaempresa.com] — a configurar.

FAQ — 4 Objeções Principais Extraídas dos Error Handlers

  • Como integro com meu CRM atual? Conectamos nativamente com HubSpot via API (hubspot.service.ts). Para outros CRMs, nosso SDK público (packages/sdk) e webhooks permitem integração em qualquer sistema com suporte a HTTP. Configuração em menos de 30 minutos.
  • Meus dados de prospecção são seguros? Seus dados são segregados por organização com isolamento em nível de banco de dados — nenhum outro cliente acessa suas informações. Cabeçalhos de segurança HTTP ativos (Helmet), CORS restrito e trilha de auditoria completa de todas as ações. Dados de notas sensíveis enviados ao HubSpot são criptografados antes da transmissão.
  • O sistema aguenta nosso crescimento? A arquitetura foi projetada para escala incremental: filas assíncronas com BullMQ para picos de processamento, cache distribuído para latência consistente e banco PostgreSQL com índices otimizados por organização. Workers de enriquecimento e RAG escalam independentemente da API principal.
  • E se minha equipe precisar de ajuda? A API é totalmente documentada em /api/docs (Swagger/OpenAPI). Nosso SDK público tem exemplos prontos. O plano Enterprise inclui onboarding técnico, Customer Success dedicado e SLA de suporte contratual.
